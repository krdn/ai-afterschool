# Phase 9: Performance & Database Optimization - Context

**Gathered:** 2026-01-30
**Status:** Ready for planning

## Phase Boundary

데이터베이스 성능 최적화, 연결 풀링, 이미지 최적화를 통해 프로덕션 규모(50~200명 학생)에서 안정적인 성능을 제공하는 인프라 구축

## Implementation Decisions

### 데이터베이스 마이그레이션 정책
- 배포 시 `prisma migrate deploy`가 자동으로 실행되어 적용되지 않은 마이그레이션을 자동 적용
- 마이그레이션 실패 시 즉시 배포를 중단하고 이전 버전으로 자동 롤백
- 짧은 다운타임 허용: 마이그레이션 실행 동안 애플리케이션을 잠시 중단하고 완료 후 재시작
- 마이그레이션 전후에 데이터베이스 전체 백업을 자동으로 생성 (항상 전체 백업)
- 마이그레이션 실행 결과를 모두 알림 (Slack, 이메일 등 + 배포 로그 기록)
- 마이그레이션 파일 자체가 문서이므로 별도 기록 없음
- 마이그레이션 자체 검증만 하고 추가 테스트는 없음

### 연결 풀링 한도 설정
- 고정 값 사용: 최대 10개 연결 (작은 규모, 빠른 응답)
- 연결 풀이 고갈되면 요청을 큐에 넣고 연결이 반환될 때까지 대기 (기본 60초)
- 환경별로 다름: 프로덕션과 개발 환경에서 다른 설정 사용 (개발에서는 더 적게)
- 연결 풀 사용률을 주기적으로 기록하고 경보 설정

### 쿼리 최적화 범위
- 모든 N+1 쿼리를 식별하고 한 번에 수정 (완전한 해결)
- 자동으로 감지된 N+1 쿼리만 수정 (Prisma 로그 또는 도구 사용)
- 데이터베이스 인덱스는 실제로 느린 쿼리(EXPLAIN 분석)에서만 추가
- 특별한 목표 없이, 발견되는 대로 최적화

### 이미지 최적화 전략
- Next.js Image 컴포넌트에 위임: 원본을 유지하고 Next.js Image 컴포넌트가 최적화하여 제공
- 썸네일/원본 분리: 업로드 시 자동으로 썸네일과 원본을 분리하여 저장
- 이미지 품질-크기 트레이드오프는 균형 있는 품질 (60-80)으로 파일 크기와 화질 균형
- 이미지 캐시 정책은 단기 캐시 (1시간~1일) 하여 변경을 빠르게 반영
- Cloudinary를 계속 사용 (자동 최적화, CDN 포함)

### Claude's Discretion
- 마이그레이션 실행 시간 최적화
- 연결 풀 대기 타임아웃 구체 값
- N+1 쿼리 감지 도구 선택
- 이미지 썸네일 크기 기준
- 캐시 헤더 구체 설정
